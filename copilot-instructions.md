# Copilot Workspace Instructions（ABAP 合规开发）

本文件用于让 Copilot 在本工作区内默认遵守 ABAP 合规编码流程。

注意（强制）：在对任何 ABAP 对象进行修改或生成 ABAP 代码之前，必须先读取并遵守 `clean-abap` 的 SKILL.md。

## 总体目标

- 产出代码必须：可语法检查、可激活、可运行、可查出数据。
- 优先解决“能跑通/能出数”，再做美化（列头、顺序、扩展字段）。

## 三条硬规则（必须）

1. **不允许猜测**：不猜 DDIC 字段/文本表/参数名。遇到“字段不存在/方法不存在”必须回到系统真实对象校验后再改。
2. **激活才算完成**：对 ABAP 对象的修改仅在成功激活后才算生效。
3. **先保证能出数**：选择屏幕必须支持用最少必输条件查出至少 1 条数据。

## 注释要求（必须）

- 注释只写在流程节点、关键判断、复杂 JOIN/过滤、回填测试数据、权限校验这些地方。
- 注释必须说明“为什么这么做”，不要重复代码字面意思。
- ABAP 新增注释统一用中文，简短明确，避免长段落。
- 变量声明和 DDIC 类型声明也要加短注释，例如 `data gv_bukrs type bukrs.` 后面说明“公司编码选择条件”，`type knvv-vkorg` 后面说明“销售组织”。
- 选择屏幕定义也要加行尾注释，例如 `select-options s_bukrs for gv_bukrs.` 后面说明“公司编码筛选条件”，`parameters p_vkorg type knvv-vkorg obligatory.` 后面说明“销售组织，必输”。
- 如果是为了防止误过滤数据，必须在注释里明确说明哪一段条件是固定条件，哪一段是可选条件。
- 如果改动了测试数据回填逻辑，必须写清楚它会回填哪些字段、为什么能出数。

## 文件修改权限（必须）

- **禁止修改非 kn681 用户的文件/对象**：仅允许修改由 kn681 创建/维护的代码与文档；若不确定归属，先停止并向用户确认。

## 开发完成后的测试（必须）

- **写完代码必须测试**：交付前必须在系统中实际执行一次（或多次）验证。
- **必须找出可出数的选择屏幕输入**：需要通过数据库查询/样例数据，给出一套（至少一套）满足程序固定条件与业务逻辑的输入组合（如 VKORG/BUKRS/KUNNR/日期范围等），并用该输入跑出真实数据。
- **无数据不算完成**：若测试输入仍无结果，必须回到查询条件/JOIN/选择屏幕默认值进行排查，直到能稳定出数。

## 交付标准（必须）

- 修改 ABAP 对象后必须完成语法检查和激活。
- 若是报表或查询程序，必须说明至少一套可出数的测试条件。
- 若是 ALV 输出，必须确认列头和关键字段顺序没有被意外打乱。
- 若是自定义指令文件，必须保持 Markdown 结构清晰，不要把说明写成散乱条目。

## ABAP FS（adt://mysap）工具与操作约束

- 在 `adt://mysap` 虚拟文件系统中：不要用本地文件系统的 `grep/find` 思路；搜索 ABAP 代码优先用 ABAP 专用搜索工具（如 `search_abap_object_lines`）。
- 修改 ABAP 对象后：做语法检查（Problems）→ 激活对象。

## 需求→设计→实现的强制步骤

### 1) 需求拆解

- 明确：选择屏幕字段（必输/可选/范围）、固定业务条件（例如 VTWEG/SPART 是否固定）、输出字段映射（字段→表字段/文本表字段→连接条件）、权限对象。

### 2) 查询设计（避免 0 数据）

- 主查询先做“最小可出数版本”：只 INNER JOIN 必须存在的表；文本表/描述表一律 `LEFT OUTER JOIN` + `SPRAS = sy-langu`。
- 银行/合作伙伴/税号等扩展信息：优先二次查询 + HASHED MAP 补齐，避免 JOIN 把主数据过滤没。

### 3) 选择屏幕必须提供“测试数据回填”

- 必须提供一个功能键：自动从数据库挑 1 条符合固定条件的记录，把关键字段回填到选择屏幕（例如 VKORG、BUKRS、KUNNR、宽范围 ERDAT）。
- 回填时清空其它可选过滤（NAME/属性/业务员等），防止把数据过滤掉。

### 4) ALV 输出

- 优先 `CL_SALV_TABLE`，开启标准功能（排序/筛选/导出）。
- 列头文本用中文显式设置；如果系统 SALV 不支持列位置方法，则通过输出结构字段顺序控制列顺序。

### 4.1) 注释写法

- 头部注释说明程序目的、主要输入、主要输出、关键依赖表。
- 主流程注释说明执行顺序，例如“先校验权限，再取数，再显示 ALV”。
- 复杂取数逻辑要标明“主数据查询”和“补充信息查询”两个阶段。
- 回填测试数据要标明会清空哪些筛选条件，避免把数据再次过滤掉。
- 声明区注释要贴近字段本身，写成“字段用途”而不是重复类型名。
- 选择屏幕注释要贴近筛选项本身，写成“筛选条件用途”而不是重复语句本身。

### 5) 权限与消息

- 程序入口处做 `AUTHORITY-CHECK`（按需求对象）。
- 无权限用 `MESSAGE E`；无数据用 `MESSAGE S` 并退出。

## Debug / 排错（查不出数据时）

按顺序定位，不靠猜：

1. 单表计数（验证固定条件是否有数据）
2. 主查询计数（逐个 JOIN/WHERE 找出是哪一步把数据过滤没）
3. 放宽可选 JOIN 为 LEFT OUTER JOIN
4. 校验语言字段（SPRAS）与文本表
