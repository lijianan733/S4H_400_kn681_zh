# 用户角色差异分析及权限补齐工具 需求分析

## 1. 文档目的

本文档用于沉淀“用户角色差异分析及权限补齐工具”的业务需求、功能边界、交互流程和技术实现要点，供顾问评审、ABAP 开发、测试验收共同使用。

本工具目标不是简单复制用户，而是基于两个已存在用户的角色集合进行差异分析，并在分析结果基础上执行授权补齐动作，满足权限复用和权限治理场景。

## 2. 业务背景

当前业务存在以下典型诉求：

1. 需要判断用户 A 与用户 B 的角色差异。
2. 需要让用户 A 至少拥有用户 B 的全部角色。
3. 用户 A 原本已有的额外角色需要保留，不能被误删。
4. 需要一个可复用工具，而不是只针对单一“复制 B 到 A”的一次性程序。
5. 业务希望先查看差异结果，再由操作人员在结果界面确认并执行动作，而不是在选择屏幕直接更新权限。

## 3. 建设目标

建设一个 ABAP 报表工具，实现以下目标：

1. 输入两个已存在的 SAP 用户 A 和 B。
2. 比较 A、B 当前有效角色。
3. 将比较结果分组展示在 ALV 中。
4. 支持在 ALV 结果界面基于选中行触发动作。
5. 第一阶段重点支持“将 B 有但 A 没有的角色补充到 A”。
6. 工具结构可扩展，后续可以增加“补充到 B”“删除候选”“双向同步”等能力。

## 4. 适用范围

### 4.1 当前范围

1. 仅处理已存在的 SAP 用户。
2. 仅处理用户角色分配，不直接处理授权对象明细。
3. 仅比较角色名级别的数据。
4. 默认比较当前有效角色。
5. 第一版不做角色删除。
6. 第一版先实现单组用户 A/B 的比较与执行。

### 4.2 暂不纳入范围

1. 不处理批量多用户对比。
2. 不处理授权对象级别的精细化比对。
3. 不处理派生角色展开比对。
4. 不处理直接授权对象差异。
5. 不自动覆盖角色有效期差异。

## 5. 关键业务定义

### 5.1 角色差异分类

系统需将角色比较结果划分为以下三类：

1. A没有，B有
   - 说明：角色存在于 B，不存在于 A。
   - 业务意义：可作为“补充到 A”的候选集。

2. A有，B没有
   - 说明：角色存在于 A，不存在于 B。
   - 业务意义：用于分析差异，后续可扩展为补充到 B 或治理动作。

3. A、B都有
   - 说明：角色同时存在于 A 与 B。
   - 业务意义：作为共同角色分析结果展示，不参与第一版执行动作。

### 5.2 “至少包含”定义

“A 至少包含 B”定义为：

1. B 当前拥有的全部角色，A 最终都应拥有。
2. A 可以拥有额外角色。
3. 不要求 A 与 B 完全一致。

数学上可理解为：B 的角色集合是 A 的子集。

## 6. 操作流程

### 6.1 用户操作主流程

1. 操作人员在选择屏幕输入用户 A 和用户 B。
2. 选择需要查看的差异范围。
3. 执行报表，系统读取 A/B 角色并完成比对。
4. 系统以 ALV 方式展示比较结果。
5. 操作人员在 ALV 中选择目标记录。
6. 操作人员点击 ALV 工具栏上的“执行动作”按钮。
7. 系统根据当前选中结果执行对应动作。
8. 系统返回执行状态并刷新 ALV 结果。

### 6.2 交互原则

1. 选择屏幕只负责查询和比对条件输入。
2. 授权动作必须在 ALV 结果界面中触发。
3. 不允许在未查看结果前直接执行权限变更。
4. 先展示、再执行，降低误操作风险。

## 7. 选择屏幕需求

### 7.1 输入项

1. 用户 A
2. 用户 B
3. 差异显示范围，建议支持：
   - 仅显示 A没有B有
   - 仅显示 A有B没有
   - 仅显示 A、B都有
   - 显示全部
4. 是否仅比较当前有效角色
5. 是否测试运行（供后续扩展使用）

### 7.2 选择屏幕职责

1. 校验用户是否存在。
2. 校验 A 与 B 不可为空。
3. 校验 A 与 B 不建议相同。
4. 不在选择屏幕执行授权变更。

## 8. ALV 展示需求

### 8.1 ALV 字段建议

1. 选择标识
2. 角色名称
3. 分类
4. 用户 A
5. 用户 B
6. 建议动作
7. 执行状态
8. 返回消息

### 8.2 ALV 按钮需求

第一版建议至少支持以下按钮：

1. 执行选中行
2. 刷新结果
3. 导出结果

后续可扩展：

1. 执行全部候选
2. 补充到 B
3. 删除候选角色

## 9. 第一版动作规则

### 9.1 第一版允许执行的动作

第一版建议仅正式开放以下动作：

1. 对分类为“A没有，B有”的选中记录执行“补充到 A”。

### 9.2 第一版只分析不执行的内容

以下内容在第一版只做展示，不做正式执行：

1. A有，B没有
2. A、B都有
3. 删除角色
4. 双向同步

### 9.3 原因说明

第一版聚焦“把 B 的缺失角色补给 A”，这是当前最核心、最安全、最明确的业务诉求。删除和双向同步会带来更高风险，应放在后续版本。

## 10. 数据口径

### 10.1 比较口径

1. 以角色名为主键比较。
2. 仅取当前有效角色参与比较。
3. 不展开角色中的授权对象明细。
4. 不比较组织级授权字段值。

### 10.2 角色来源

角色来源需读取 SAP 标准用户角色分配数据，建议以用户当前角色分配关系作为基础来源。

## 11. 技术设计建议

### 11.1 整体结构

建议采用“可执行报表 + 本地类服务”的方式实现。

### 11.2 职责划分

#### 报表层职责

1. 处理选择屏幕。
2. 调用比较服务。
3. 展示 ALV。
4. 响应 ALV 用户命令。

#### 服务类职责

1. 读取用户角色。
2. 比较角色集合。
3. 生成 ALV 结果集。
4. 执行授权补齐动作。
5. 回写执行状态与消息。

### 11.3 建议方法划分

1. `run`
   - 主流程控制。
2. `validate_input`
   - 校验输入参数。
3. `read_user_roles`
   - 读取指定用户角色。
4. `compare_roles`
   - 生成三类差异结果。
5. `build_alv`
   - 构建 ALV 展示所需数据。
6. `display_alv`
   - 展示 ALV。
7. `execute_selected_actions`
   - 执行 ALV 选中记录的补齐动作。
8. `refresh_result`
   - 执行后刷新差异结果。

## 12. 授权更新策略

### 12.1 基本原则

1. 不直接更新授权相关数据库表。
2. 应通过标准用户维护接口/BAPI完成角色分配。
3. 执行后需要提交事务并记录返回消息。

### 12.2 第一版执行规则

1. 只对选中且分类为“A没有，B有”的行执行。
2. 若角色已存在，则跳过。
3. 执行后返回成功/失败消息。
4. 刷新后该角色应转为“A、B都有”或从当前过滤结果中消失。

## 13. 风险与控制点

### 13.1 业务风险

1. 高权限角色可能被错误补齐。
2. 操作人员可能误选过多角色。
3. A/B 用户角色有效期可能存在业务差异。

### 13.2 技术风险

1. 标准接口对角色维护字段格式要求严格。
2. 角色分配后可能需要重新读取结果才能正确展示。
3. 复合角色和单角色混用时，显示与业务认知可能不完全一致。

### 13.3 控制措施

1. 先展示、后执行。
2. 只执行选中行。
3. 第一版先不支持删除。
4. 对执行结果做状态与消息反馈。
5. 建议保留操作日志。

## 14. 验收标准

### 14.1 查询验收

1. 输入 A/B 后能够正确读取双方角色。
2. 能准确区分三类结果。
3. ALV 展示内容完整、分类正确。

### 14.2 执行验收

1. 对“A没有，B有”的选中结果能成功补充到 A。
2. A 原有额外角色不受影响。
3. 执行后刷新结果正确。
4. 失败记录有明确消息。

### 14.3 边界验收

1. A、B 任一不存在时有错误提示。
2. A 与 B 相同时有明确提示。
3. 无差异时 ALV 结果应有清晰反馈。
4. 未选中任何记录时不允许执行。

## 15. 后续扩展建议

1. 支持“把 A有B没有 的角色补给 B”。
2. 支持“删除角色”但需二次确认。
3. 支持批量 A/B 对比。
4. 支持导出差异分析结果。
5. 支持更细粒度的授权对象差异分析。
6. 支持操作日志落表。

## 16. 当前结论

本需求第一阶段应聚焦为：

1. 生成用户 A 与用户 B 的角色差异分析结果。
2. 在 ALV 结果界面支持对“A没有，B有”的选中角色执行“补充到 A”。
3. 保证交互安全、实现可扩展、结构可复用。

该方案既满足当前“让 A 至少包含 B”的核心诉求，也为后续扩展成完整的角色对比和同步工具保留了清晰的演进路径。
