# 航班查询报表实现规范（跨 AI 通用版）

## 1. 文档目的
这份文档不是某一个 AI 的专用记录，而是团队统一的工程规范。
无论使用哪种 AI 助手，只要遵循本文件，代码实现路径、质量标准和调试流程应保持一致。

## 2. 任务范围
1. 在现有报表 Z260403_REPORT_1 上实现 OOP 航班查询。
2. 数据来源为 SFLIGHT。
3. 结果展示为 REUSE_ALV_GRID_DISPLAY_LVC。
4. 支持常见语法版本并完成激活验证。

## 3. 统一实现流程
1. 读取当前源码并确认对象可编辑。
2. 明确输入、输出和边界条件。
3. 先设计类结构，再写代码。
4. 建立查询模型与结果结构，保证字段类型一致。
5. 分离职责：查询、字段目录、展示分别放在独立方法。
6. 小步修改，小步激活，小步验证。
7. 每次修改后检查 Problems，直到清零。

## 4. OOP 结构要求
建议采用一个主类 lcl_report，至少包含以下方法：
1. run：流程编排。
2. read_data：执行查询。
3. build_fieldcat：构建 ALV 字段目录。
4. display_alv：调用 ALV 展示。

要求：
1. 方法职责单一。
2. 查询逻辑和展示逻辑分离。
3. 错误信息在流程层统一处理。

## 5. 编码规范（跨 AI 统一）
1. 禁止无差别 SELECT *，只查需要字段。
2. 结果内表行类型必须与 SELECT 列严格对齐。
3. 尽量使用可读性高的命名和单一职责方法。
4. 对系统版本不确定时，优先使用稳定语法而非新语法糖。
5. ALV 相关类型使用前先确认类型池或类型可见性。

## 6. 调试与修复流程（标准作业）
1. 先看 Problems 报错，记录行号与报错文本。
2. 先修类型错误，再修语法兼容，再修运行逻辑。
3. 每修复一类问题立即激活，不要堆积改动。
4. 若报错来自系统标准对象，先排除，不要误改标准对象。
5. 最终以目标对象激活成功且 Problems 清零为完成条件。
6. Debug 结束必须向 DEBUG_SESSION_LOG.md 追加一条记录。

## 7. 本次典型问题与通用解法
1. 查询结果类型不匹配：
   - 现象：字段类型冲突。
   - 解法：定义本地结构，字段顺序与类型和 SELECT 完全一致。
2. RTTI 方法不可用：
   - 现象：方法未知或权限不可见。
   - 解法：改用更兼容的类型描述路径，避免依赖高版本专用调用。
3. inline 声明在旧语法下异常：
   - 现象：局部变量被识别为未定义。
   - 解法：改用经典 DATA 声明方式。
4. LVC 类型识别失败：
   - 现象：字段目录或布局类型不可识别。
   - 解法：补全类型池声明并重新激活。

## 8. 验收标准
1. 程序可激活。
2. Problems 对目标程序无错误。
3. 空条件、单条件、多条件都可运行。
4. ALV 正常显示且字段含义正确。
5. 无数据时有清晰提示。

## 9. 后续增强建议
1. 增加选择屏幕文本元素。
2. 增加权限检查和审计信息。
3. 增加结果上限和性能保护。
4. 增加可测试层，便于单元测试。

## 10. 给未来 AI 的执行提示
1. 先读本文件，再读项目约束文件。
2. 先给实现计划，再落地代码。
3. 每次改动后必须激活和复检。
4. 不跳过错误定位步骤，不跳过验证步骤。

## 11. 弱 AI 强约束协议（必须执行）
适用于能力较弱、容易跳步或容易猜测的 AI。

1. 禁止直接编码，先输出以下四项：
   - 需求复述
   - 输入输出定义
   - 风险点列表
   - 5 步以内实施计划
2. 未确认前不得进入编码。
3. 编码时每次只改一个问题域：
   - 类型问题
   - 语法问题
   - 运行逻辑问题
4. 每次改动后必须执行：
   - 激活目标对象
   - 读取 Problems
   - 输出本轮变更与结果
5. 禁止猜测 API、方法名、字段名；不确定时先查询再写。
6. 如果出现标准对象报错，先与目标对象报错分离，禁止误修系统标准对象。

## 12. 标准 Debug 回路（循环直到通过）
1. 收集错误：读取目标对象 Problems。
2. 分类错误：按类型/语法/逻辑分组。
3. 选择最上游错误优先修复。
4. 小步修改并记录修改点。
5. 立即激活对象。
6. 再次读取 Problems。
7. 若仍有错误，回到步骤 2。
8. 通过条件：目标对象激活成功且 Problems 清零。

## 13. 交付输出格式（所有 AI 统一）
每次任务结束时，输出应包含：
1. 变更文件清单
2. 关键改动说明
3. 验证动作与结果
4. 未解决风险与下一步建议
5. 已写入 DEBUG_SESSION_LOG.md 的记录标题
