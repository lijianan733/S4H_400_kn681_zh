# 用户角色差异分析及权限补齐工具

## 一、需求分析

### 1.1 目标
实现一个 ABAP 报表，用于输入两个已存在用户 A/B，比较角色差异，展示结果，并支持按差异方向补全到 A 用户或 B 用户。

### 1.2 业务目标
- A 至少包含 B，即 B 的角色集合应为 A 的子集。
- A 原有额外角色必须保留，不做删除。
- 支持将 B 有但 A 没有的角色补全到 A 用户。
- 支持将 A 有但 B 没有的角色补全到 B 用户。

### 1.3 比较范围
- 仅处理已存在用户。
- 仅按角色名比较。
- 仅单组 A/B 对比。
- 不直接更新授权表。

### 1.4 差异分类
系统将角色分为三类：
- A没有，B有：补全到 A 用户的候选。
- A有，B没有：补全到 B 用户的候选。
- A、B都有：共同角色，仅展示。

### 1.6 验收标准
- 能正确读取 A/B 角色并区分三类结果。
- 对“A没有，B有”的选中记录能成功补全到 A 用户。
- 对“A有，B没有”的选中记录能成功补全到 B 用户。
- A 原有额外角色不受影响。
- 无差异、未选中、用户不存在、A/B 相同等场景都有明确提示。

## 二、界面功能

### 2.1 选择屏幕
- 用户 A：接收被补全目标用户。
- 用户 B：接收参照用户。
- 差异显示范围：
	- 仅 A没有B有
	- 仅 A有B没有
	- 仅 A、B都有
	- 全部
- 测试运行：仅做模拟校验，不提交权限变更。
- 校验：A/B 非空、存在，且不建议相同。
- 职责：只做条件输入与校验，不执行权限变更。

### 2.2 ALV 展示字段
- 选择标识：勾选后参与执行。
- 角色名称：显示差异角色名。
- 分类：显示 A没有B有 / A有B没有 / A、B都有。
- 用户 A：显示该角色在 A 用户中的状态。
- 用户 B：显示该角色在 B 用户中的状态。
- 建议动作：显示系统建议执行的动作，如“补全到 A 用户”“补全到 B 用户”“仅展示”。
- 执行状态：显示未执行 / 成功 / 失败 / 跳过。
- 返回消息：显示执行或校验消息。

### 2.3 ALV 按钮规则
- 按差异范围动态显示按钮。
- 仅显示“A没有，B有”时：补全到A用户、刷新结果、导出结果。
- 仅显示“A有，B没有”时：补全到B用户、刷新结果、导出结果。
- 仅显示“A、B都有”时：显示刷新结果、导出结果。

## 三、后台功能实现

### 3.1 主流程
1. 选择屏幕输入 A、B、差异范围、测试运行标志。
2. 校验 A/B 非空、存在，且不建议相同。
3. 报表读取 A/B 角色并完成比对。
4. ALV 展示结果。
5. 用户在 ALV 中选择记录并点击执行。
6. 系统按记录分类执行补全到 A 用户或补全到 B 用户，并刷新结果。

### 3.2 技术方案
- 采用“可执行报表 + 本地服务类”。
- 服务类职责：读取用户角色、比较集合、构建 ALV 数据、执行补齐、刷新结果并回写消息。
- 关键方法：`run`、`validate_input`、`read_user_roles`、`compare_roles`、`build_alv`、`display_alv`、`execute_selected_actions`、`refresh_result`。

### 3.3 授权更新原则
- 不直接更新数据库表。
- 通过标准用户维护接口/BAPI 完成角色分配。
- 执行后必须提交事务并返回消息。

### 3.4 风险与控制
- 风险：高权限角色误补、误选过多、角色有效期差异。
- 控制：先展示后执行、只执行选中行、保留操作日志、显示明确反馈。
